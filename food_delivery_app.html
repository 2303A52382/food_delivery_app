Zomato Web Application — Modular Design

This document provides a modular breakdown tailored to a Zomato-like web application: modules, interfaces, design rationale, and a UML component diagram rendered as SVG.

Primary modules (low coupling, single responsibility)

Auth & User Module

  • Registration/login (OAuth, email, phone), profile, saved addresses, KYC for partners
  • APIs: AuthService, UserProfileService
  • Depends on: Notification, DB

Restaurant Partner Module

  • Partner onboarding, menu management, order acceptance dashboard, operating hours
  • APIs: PartnerPortalAPI, MenuMgmtService
  • Depends on: MediaStore, Billing

Catalog, Menus & Collections

  • Restaurant discovery, collections (trending, offers), cuisines, photos
  • APIs: CatalogAPI, CollectionsService
  • Depends on: Search, Cache

Search & Personalization

  • Text/geo search, faceted filters, personalized recommendations (taste, past orders)
  • APIs: SearchAPI, RecoService
  • Depends on: Analytics, Catalog

Order & Cart Module

  • Cart, checkout, order lifecycle, invoice generation
  • APIs: OrderService, CartService
  • Depends on: Payment, PartnerModule, Promo

Payments & Wallet

  • Multiple gateway integrations, Zomato Wallet, refunds, settlements to partners
  • APIs: PaymentGateway, WalletService
  • Depends on: Order, Accounting

Delivery & Tracking

  • Assign riders, live tracking, ETA, route optimization, COD support
  • APIs: DeliveryService, TrackingAPI
  • Depends on: Maps API, Event Bus

Promotions & Offers (Zomato Pro)

  • Memberships, targeted offers, coupon engine, stacking rules
  • APIs: PromoEngine, SubscriptionService
  • Depends on: Billing, Analytics, User

Reviews, Ratings & Content

  • Reviews, photos, editorials, content moderation, collections
  • APIs: ReviewService, ContentService
  • Depends on: User, PartnerModule, ML moderation

Admin, Reporting & Settlements

  • Dashboards, partner settlements, fraud detection, audit logs
  • APIs: AdminAPI, FinanceAPI
  • Depends on: Data Warehouse, Event Bus

Why modularity matters

  • Reusability: Services like PaymentGateway or NotifyService are reused across web, mobile, and partner integrations.
  • Maintainability: Bug fixes and feature changes are isolated. E.g., change in wallet logic doesn't force redeploy of Search.
  • Scalability: Traffic-heavy modules (Search, Catalog) can be independently scaled and optimized with caching and read replicas.
  • Operational safety: Using event-driven interactions (Kafka) reduces blast radius for failures.
  • Faster releases: Teams can own modules and deploy independently with API contracts and semantic versioning.

Implementation tips for Zomato web:
  • Serve the web client via CDN, SSR for SEO (restaurant pages), and hydrate for interactivity.
  • Cache menu and collection pages aggressively; use ES for search and personalization indices.
  • Design partner settlements in the Finance service; keep transactional money flow auditable and separate from business logic.
  • Protect partner-facing endpoints with stricter auth and role-based access control.
